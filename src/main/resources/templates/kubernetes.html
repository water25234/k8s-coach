這份教學拆成多頁。先看 Build 到 Deploy 和部署流程，再理解 image、command、kubelet， 最後逐頁讀 Namespace、Deployment、Service 的 YAML。每一頁都會把本專案實際檔案和用途對起來。

從檔案到部署的主線流程

這個專案不是直接把 Java 丟進 Kubernetes。真正流程是：先用 Dockerfile 把 Spring Boot 打包成 Docker image，再讓 Docker Compose 或 Kubernetes 使用同一個 image。

1

Spring Boot 原始碼

src/main/java 和 src/main/resources/templates 是網站程式與 HTML 頁面。

2

Dockerfile 建立 image

Dockerfile 先用 Maven build JAR，再把 JAR 放進 Java 21 runtime image，最後產生 deployable-spring-site:0.1.0。

3

docker-compose.yml 本機跑 container

docker compose up --build 會使用同一份 Dockerfile build image，然後用 ports: 8080:8080 讓瀏覽器直接打 localhost。

4

deployment.yaml 讓 K8s 跑 Pod

image: deployable-spring-site:0.1.0 指向前面 build 出來的 image，replicas: 2 讓 Kubernetes 跑兩個 Pod。

5

service.yaml 提供 Pod 入口

Pod 會重建、IP 會變，所以 Service 用 selector.app 找 Pod，提供穩定的 cluster 內部入口。

6

port-forward 讓本機瀏覽器進入 K8s

因為這個 Service 是 ClusterIP，本機要看 K8s 裡的網站，需要用 kubectl port-forward service/deployable-spring-site 8080:8080 -n spring-demo。

Dockerfile

定義 image 怎麼被做出來。

docker-compose.yml

本機用 Docker 跑同一個 image。

k8s/deployment.yaml

Kubernetes 用同一個 image 建 Pod。

k8s/service.yaml

Kubernetes 幫 Pod 建立穩定入口。

K8s 架構總覽

這張圖說明 Kubernetes cluster 裡有哪些角色。它不是瀏覽器流量圖，而是 Kubernetes 如何接收 YAML、決定 Pod 放哪裡、並讓 Worker Node 執行 container 的管理架構。

Control Plane

負責管理整個 cluster，也就是你圖片裡的 Master。Master 是舊稱，現在官方文件常稱 Control Plane。

API Server kubectl apply 會先打到這裡，所有 YAML 都由它接收。
Scheduler 決定新的 Pod 要放到哪一台 Worker Node。
Controller Manager 持續檢查實際狀態，例如 Deployment 是否真的有 2 個 Pod。
etcd 儲存 cluster 狀態，例如 Namespace、Deployment、Service 設定。

Worker Nodes

真正執行你的 Spring Boot container 的機器。

Worker Node A kubelet 接收 Pod 任務，請 Container Runtime 啟動 container。 Pod A：Spring Boot
Worker Node B kube-proxy 協助 Service 把流量導到 Pod。 Pod B：Spring Boot
你貼的圖是在講管理流程

kubectl 把 YAML 送到 API Server，Control Plane 決定如何建立 Pod，Worker Node 負責真的執行 container。

下面的圖是在講網站流量

瀏覽器不是直接打 API Server，而是透過 port-forward 連到 Service，再由 Service 轉到 Pod。

本專案網站流量

本專案的 Service 是 ClusterIP，所以瀏覽器不能直接連到 cluster 內部。 本機測試時要先用 kubectl port-forward 建立臨時通道。

1

你的瀏覽器

你在本機打開 http://localhost:8080/kubernetes，瀏覽器送出 HTTP request。

2

kubectl port-forward

因為 Service 是 ClusterIP，外部瀏覽器不能直接連進 cluster，所以用 kubectl port-forward 把本機 8080 暫時接到 Service 8080。

3

Service

Service 讀取 selector.app: deployable-spring-site，尋找 label 符合的 Pod，並把流量轉送過去。

4

Pod A / Pod B

Deployment 設定 replicas: 2，所以 Kubernetes 會維持兩個 Pod。每個 Pod 裡都跑同一個 Spring Boot container。

5

Spring Boot

Spring Boot 處理 /、/api/status、/kubernetes 等路由，回應再沿原路回到瀏覽器。

三個 YAML 檔案如何接起來

1. namespace.yaml 建立 spring-demo

它只建立一個命名範圍，不會啟動網站。

2. deployment.yaml 放進 spring-demo

metadata.namespace: spring-demo 讓 Deployment 進入剛建立的 Namespace，並建立 Pod 來跑 Spring Boot。

3. service.yaml 也放進 spring-demo

Service 用 selector.app: deployable-spring-site 找到 Deployment 建出的 Pod。

4. port-forward 讓本機瀏覽器連進 Service

因為 Service 是 ClusterIP，本機要看頁面就需要 kubectl port-forward。

手動操作指令

第一次部署時使用這組指令。它會建立 Namespace、Deployment、Service，最後用 port-forward 讓瀏覽器可以透過 localhost 看到網站。

  1. 1
    建立 namespace kubectl apply -f k8s/namespace.yaml

修改網站後重新部署

如果你改了 Java、HTML 或 CSS，Kubernetes 裡的 Pod 不會自動拿到新版程式。 你需要先重新 build image，再要求 Deployment 重啟 Pod，最後重新建立 port-forward。

  1. 1
    重新 build Docker image docker build -t deployable-spring-site:0.1.0 .

完成後開啟 http://localhost:8080/kubernetes。如果頁面還是舊的，先確認 Docker image 是否有重新 build，並檢查 Pod 是否真的有被重啟。
