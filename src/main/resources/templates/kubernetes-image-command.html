Container Startup

image 是來源，command 是啟動指令

image 告訴 Kubernetes 要用哪個 container image 建立 container。 command 是 container 啟動後要執行什麼程式。它不會 build image，也不會把 Java 原始碼打包。

先分清楚四個名詞

Dockerfile

定義 image 怎麼被做出來，例如複製 JAR、設定使用者、設定預設啟動指令。

ENTRYPOINT ["java", "-jar", "/app/app.jar"]
image

已經 build 好的 container 包。Kubernetes 只會用 image 建 container，不會讀你的 Java 原始碼。

image: deployable-spring-site:0.1.0
command

Kubernetes 裡的 command 會覆蓋 Dockerfile 的 ENTRYPOINT。它是啟動 container 的程式。

command: ["java", "-jar", "/app/app.jar"]
args

Kubernetes 裡的 args 會覆蓋 Dockerfile 的 CMD。常用來補啟動參數。

args: ["--server.port=8080"]

Container 啟動 system flow

這張流程是解釋「image 和 command 何時被使用」，不是解釋瀏覽器流量。

1

Dockerfile build image

docker build 讀 Dockerfile，把 Spring Boot JAR 放進 image，並設定預設 ENTRYPOINT。

2

image 存在某個地方

本機學習可能在 Docker Desktop；正式環境通常在 Docker Hub、GHCR、ECR、GCR 或 ACR。

3

Deployment 指定 image

deployment.yaml 的 image 指向已存在的 image。Kubernetes 會用它建立 container。

4

command 決定啟動方式

若 YAML 沒寫 command，就用 image 裡的預設 ENTRYPOINT。若有寫，就覆蓋預設啟動方式。

5

Spring Boot 開始跑

container 裡的 Java process 啟動後，才會開始聽 8080、回應 /、/api/status、/kubernetes。

本專案目前怎麼做

Dockerfile

ENTRYPOINT ["java", "-jar", "/app/app.jar"]
deployment.yaml 沒有 command

所以使用 Dockerfile 的 ENTRYPOINT

本專案的 Deployment 只寫 image，沒有寫 command。因此 container 啟動時會使用 Dockerfile 裡的 ENTRYPOINT。

image: deployable-spring-site:0.1.0

這只是指定 image

這行不會 build Java、不會執行 Maven。它只告訴 Kubernetes 要用哪個已經 build 好的 image。

你看到的 HAPI FHIR YAML 為什麼有 command

那份 YAML 的 command 是覆蓋 container 啟動方式，不是 image，也不是 build 指令。 

containers:
  - name: hapi-fhir-server
    image: hapi-fhir-jpaserver-starter-hapi-fhir-jpaserver-start
    command:
      - "java"
      - "--class-path"
      - "/app/main.war"
      - "org.springframework.boot.loader.PropertiesLauncher"
image: ...

真正的 image 在這裡

Kubernetes 會先用這個 image 建 container。這個 image 內必須已經有 /app/main.war。

command: ["java", ...]

啟動 container 後跑 java

這段只是 container 啟動時執行的程式。如果 image 裡沒有 /app/main.war，Pod 會啟動失敗。

初學者判斷規則

看到 image

問自己：這個 image 在哪裡？本機 Docker？Minikube？Kind？還是 registry？Kubernetes 拿得到嗎？

看到 command

問自己：這是在 container 啟動後跑什麼？它不是 build 步驟，也不是 Dockerfile。

看到 Dockerfile

問自己：這份檔案會在 docker build 時被讀取，用來做出 image。

看到 rollout restart

問自己：Pod 會重建，但它重建時拿到的是新版 image 還是舊版 image？
