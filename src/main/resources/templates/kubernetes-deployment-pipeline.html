Deployment Pipeline

從 git push 到 Kubernetes 上線

真正的部署不是把 Java 原始碼丟給 Kubernetes。常見做法是先測試、build Docker image、推到 image registry， 再讓 Kubernetes 用新的 image tag 建立新版 Pod。

先記住四個角色

CI

負責檢查程式碼，例如跑測試、編譯、build Docker image。

mvn test / docker build
Image Registry

負責保存 image，例如 Docker Hub、GitHub Container Registry、ECR、GCR、ACR。

docker push registry/app:tag
CD

負責把新版 image 部署到 Kubernetes，例如 kubectl、Helm、Argo CD。

kubectl apply / helm upgrade / argo cd sync
Kubernetes

負責拉 image、建立 Pod、健康檢查、滾動更新。它不負責從 Java 原始碼 build image。

image: k8s-coach:0.1.0

本機手動部署流程

這是你目前學習用的流程。所有事情都在你的筆電上手動完成，適合理解概念，但不適合多人正式上線。

1

修改 Java / HTML / CSS

你改的是專案原始碼，例如 src/main/java、templates、static。

2

執行測試

確認 Spring Boot route、API、健康檢查都正常。你本機沒有 Maven 時，可以用 Docker 裡的 Maven 跑。

docker run --rm -v "$PWD:/workspace" -w /workspace maven:3.9.9-eclipse-temurin-21 mvn test
3

build Docker image

Dockerfile 會把 Spring Boot 專案打包成 image。Kubernetes 後面只會用這個 image，不會再讀你的原始碼。

docker build -t k8s-coach:0.1.0 .
4

套用 K8s YAML

deployment.yaml 指定 image，service.yaml 提供 Pod 入口。套用前要先確認你的 cluster 拿得到這個 image。

kubectl apply -f k8s/namespace.yaml -f k8s/deployment.yaml -f k8s/service.yaml
5

等待 Deployment 就緒

Kubernetes 會建立 Pod，並透過 readiness / liveness probes 判斷服務狀態。

kubectl rollout status deployment/k8s-coach -n k8s-coach
6

本機瀏覽器測試

因為本專案 Service 是 ClusterIP，要用 port-forward 暫時接到本機。

kubectl port-forward service/k8s-coach 8080:8080 -n k8s-coach

本機 cluster 到底去哪裡拿 image

這是初學者最容易卡住的地方。docker build 建好的 image 不一定等於 Kubernetes 看得到的 image。 要看你用的是哪一種 cluster。

Docker Desktop Kubernetes

通常跟本機 Docker image store 比較接近，所以 docker build -t k8s-coach:0.1.0 . 後，K8s 較容易直接用到。

適合本專案入門教學
Minikube

Minikube 可能有自己的 Docker 環境。你需要在 Minikube 裡 build，或把 image 載入 Minikube。

minikube image load k8s-coach:0.1.0
Kind

Kind 的 node 是 container。你本機 build 的 image 不會自動出現在 Kind node 裡，通常要手動載入。

kind load docker-image k8s-coach:0.1.0
遠端 EKS / GKE / AKS

遠端 cluster 看不到你筆電本機 image。正式做法是 push 到 registry，再讓 cluster pull。

docker push registry.example.com/app:<git-sha>

CI/CD system flow

這張流程是正式部署的主線。Kubernetes 位在後半段，只負責拉 image 和執行 Pod。

1

Developer git push

工程師把 Java、HTML、CSS、YAML 變更推到 Git repository。

2

CI 測試與 build

CI 執行測試，通過後讀 Dockerfile，建立新的 image。

3

Image Registry 保存 image

CI 把 image push 到 registry，並使用唯一 tag，例如 Git SHA。

4

CD 更新部署設定

CD 更新 YAML、Helm values 或 Kustomize overlay 裡的 image tag。

5

Kubernetes rollout

Deployment 建立新版 Pod，readiness 通過後才讓 Service 導流到新版 Pod。

公司常見 CI/CD 部署流程

正式環境通常不會在工程師筆電上 build image，也不會用手動複製檔案部署。 常見做法是讓 CI/CD 系統接手，確保每次部署都有紀錄、可追蹤、可回滾。

1. git push

工程師把程式碼推到 GitHub、GitLab 或公司 Git server。這是整條 pipeline 的起點。

2. CI 跑測試

CI 會執行 mvn test、靜態檢查、安全掃描。測試失敗就停止，不會 build image。

3. CI docker build

測試通過後，CI 讀取 Dockerfile，產生新的 image。正式環境建議用 Git commit SHA、build number 或語意化版本，不要一直覆蓋同一個 tag。

4. CI docker push

CI 把 image 推到 registry，例如 registry.example.com/k8s-coach/k8s-coach:a1b2c3d。

5. CD 更新 image tag

CD 把 deployment.yaml、Helm values 或 Kustomize image tag 更新成剛剛 push 的版本。

6. 部署到 Kubernetes

CD 執行 kubectl apply、helm upgrade，或由 Argo CD 偵測 Git 變更後自動同步。

7. Kubernetes rollout

Deployment 建立新版 Pod，readinessProbe 通過後才接流量。預設 RollingUpdate 會逐步替換舊 Pod；如果 YAML 設成 Recreate，則會先停舊 Pod 再建新 Pod。

8. 驗證與回滾

如果新版失敗，可以查看 log、事件，或回滾到上一個 image tag。

deployment.yaml 在 pipeline 裡做什麼

它不 build image

deployment.yaml 不會執行 mvn package 或 docker build。這些是 CI 或本機 Docker 的工作。

它指定 image

image: k8s-coach:0.1.0 代表 Pod 要用哪個已存在的 image 啟動。

它描述 runtime 設定

例如 replicas、containerPort、env、probes、resources。這些是 container 跑起來以後的 K8s 管理設定。

它讓 rollout 可控

你可以透過 Deployment 看更新狀態、重啟 Pod、回滾版本，而不是手動管理單一 container。前提是 image tag 和部署紀錄要能追蹤。

固定 tag 與 rollout restart 的陷阱

固定 tag 容易看錯版本

k8s-coach:0.1.0 如果一直重複覆蓋，你很難從 Pod 看出它到底跑哪一次 build 出來的內容。

IfNotPresent 可能用到舊 image

imagePullPolicy: IfNotPresent 表示 node 已經有同名 image 時可能不重新拉。固定 tag 搭配這個設定，容易讓新版沒有真的進 Pod。

rollout restart 只重建 Pod

kubectl rollout restart 會讓 Pod 重建，但不保證 image 內容變新。cluster 還是必須拿得到新版 image。

正式環境用唯一 tag

例如 k8s-coach:a1b2c3d。這樣可以追蹤、比較、回滾，也比較符合 CI/CD 操作。

正式環境怎麼讓使用者進來

kubectl port-forward 適合本機測試和除錯，不是正式對外服務方式。

Ingress

常見正式入口。由 Ingress Controller 接 HTTP/HTTPS，再依 host 或 path 轉到 Service。

app.example.com -> Ingress -> Service -> Pod
LoadBalancer Service

雲端環境常用。雲平台會建立外部 Load Balancer，把流量導到 K8s Service。

Cloud LB -> Service -> Pod
API Gateway

大型系統常用於驗證、限流、路由、觀測，再轉進 cluster 內部服務。

Gateway -> Ingress/Service -> Pod
port-forward

只適合本機臨時測試。終端機關掉後通道就沒了，也不適合多人使用。

localhost:8080 -> Service 8080

常見部署指令對照

  1. 1
    本機測試 docker run --rm -v "$PWD:/workspace" -w /workspace maven:3.9.9-eclipse-temurin-21 mvn test
  2. 2
    本機 build image docker build -t k8s-coach:0.1.0 .
  3. 3
    正式環境常見 image tag docker build -t registry.example.com/k8s-coach/k8s-coach:<git-sha> .
  4. 4
    推到 registry docker push registry.example.com/k8s-coach/k8s-coach:<git-sha>
  5. 5
    手動套用 YAML kubectl apply -f k8s/namespace.yaml -f k8s/deployment.yaml -f k8s/service.yaml
  6. 6
    檢查 rollout kubectl rollout status deployment/k8s-coach -n k8s-coach
  7. 7
    查看 Pod kubectl get pods -n k8s-coach -o wide
  8. 8
    回滾 Deployment kubectl rollout undo deployment/k8s-coach -n k8s-coach
  9. 9
    Minikube 載入本機 image minikube image load k8s-coach:0.1.0
  10. 10
    Kind 載入本機 image kind load docker-image k8s-coach:0.1.0

最重要的結論

Dockerfile 定義 image 怎麼做，CI 或 docker build 真的產生 image， registry 保存 image，deployment.yaml 指定 Kubernetes 要跑哪個 image。 Kubernetes 只負責拉 image 並執行 Pod，不負責從 Java 原始碼 build image。
