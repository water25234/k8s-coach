Worker Node

kubelet 是 Worker Node 上的現場管理員

kubelet 不是你手動下指令的工具。你使用的是 kubectl。 kubelet 是每台 Worker Node 上的程式，負責接收 Kubernetes 分配的 Pod 任務，並讓 container 真的跑起來。

先分清楚 kubectl 和 kubelet

kubectl

你在 terminal 使用的指令工具。它會把你的操作送到 Kubernetes API Server。

kubectl apply -f k8s/deployment.yaml
kubelet

跑在 Worker Node 上的管理程式。它負責讓被分配到這台 Node 的 Pod 真的啟動。

Worker Node 上的背景服務
API Server

Kubernetes 的主要入口。kubectl 送出的 YAML、查詢、刪除、重啟，都會先到這裡。

kubectl -> API Server
Container Runtime

真正啟動 container 的工具，例如 containerd。kubelet 會請它用指定 image 啟動 container。

kubelet -> containerd -> container

一個 Pod 被建立時發生什麼事

1

你下 kubectl 指令

例如 kubectl apply -f k8s/deployment.yaml。這不是直接叫某台機器啟動 container，而是把需求送進 Kubernetes。

2

API Server 接收 YAML

API Server 會記錄 Deployment 想要的狀態，例如 replicas: 2 和 image: deployable-spring-site:0.1.0。

3

Scheduler 決定 Pod 放哪裡

Scheduler 會選一台 Worker Node，決定新的 Pod 要放到哪台機器上。

4

kubelet 接到任務

被選中的 Worker Node 上，kubelet 會看到這台機器應該要跑某個 Pod。

5

kubelet 請 Runtime 啟動 container

kubelet 不直接執行你的 Java 程式。它會請 Container Runtime 用 Docker image 啟動 Spring Boot container。

6

kubelet 回報狀態

你看到的 Running、Ready、重啟次數、健康檢查狀態，很多都是 kubelet 回報給 API Server 後顯示出來的。

kubelet 跟本專案 YAML 的關係

image

deployment.yaml 寫 image: deployable-spring-site:0.1.0。kubelet 會讓 runtime 使用這個 image 啟動 container。

containerPort

containerPort: 8080 告訴 Kubernetes 這個 container 裡的 Spring Boot app 會聽 8080。

readinessProbe

kubelet 會定期檢查 readiness。失敗時，Pod 暫時不會接 Service 轉進來的流量。

livenessProbe

kubelet 會定期檢查 liveness。持續失敗時，kubelet 可能會重啟 container。

初學者最容易搞混的地方

kubectl 不是 kubelet

kubectl 是你操作 Kubernetes 的 CLI。kubelet 是 Worker Node 上自動執行任務的程式。

kubelet 不決定 Pod 放哪台

Pod 放哪台是 Scheduler 決定。kubelet 是在被選中的 Node 上負責執行。

kubelet 不等於 Docker

kubelet 會管理 Pod，但真正啟動 container 的通常是 containerd 這類 Container Runtime。

kubectl get pods 看到的是回報結果

你查到的 Pod 狀態來自 Kubernetes 儲存的狀態，其中包含 kubelet 從 Worker Node 回報的資訊。
